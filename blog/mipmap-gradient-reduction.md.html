**Free Mini-Batch Gradients from Mipmap Reduction** Training a neural network inside a fragment shader runs into one annoying obstacle: there is no obvious way to average gradients across a mini-batch. A pixel only sees its own work, and Shadertoy gives you neither compute shaders nor atomic operations. This post documents a small neural network that trains in real time on Shadertoy and sidesteps the problem by repurposing the GPU's built-in *mipmap* — the texture-shrinking machinery that normally exists to avoid aliasing — as a free parallel arithmetic mean. Each parameter of the network gets a 16×16 patch of per-sample gradients in a texture, and a single texture fetch at the right resolution level returns the batched gradient — exactly, not approximately, as long as the layout is aligned to the mipmap grid. # What this is Shadertoy is an online sandbox for writing *fragment shaders*: small GLSL programs that run once per pixel and output a color. Most Shadertoy shaders draw things — fractals, raymarched scenes, demos. The shader described here does something stranger: it trains a neural network. The network's weights live in a texture. Every frame, the GPU computes a forward pass, a backward pass, and an optimizer step, then renders the current state of the network's prediction over the target it's trying to fit. Watching the page in the browser, you literally see the network learn. Training a neural network on a GPU is, of course, not unusual — every PyTorch model does this. What is unusual is doing it through the *graphics* pipeline, with no compute kernels, no atomics, no cross-thread communication, and no host-side coordination. A fragment shader is designed to compute one pixel's color, in parallel with all the others, and *not talk to its neighbors*. That constraint is what makes the design interesting: the whole training loop has to be expressed in terms of "write one value per pixel," using only the primitives that exist in the rendering pipeline. The shader gives itself four passes per frame, organized as ping-ponging textures (a standard Shadertoy idiom where a buffer reads from the previous frame's copy of itself). Buffer A holds the network's parameters. Buffer B refreshes a mini-batch of training samples. Buffer C computes per-sample gradients. The Image pass visualizes the result. # A two-paragraph neural-network primer A *multi-layer perceptron* (MLP) is the simplest kind of neural network: a stack of layers, each of which takes a vector of numbers and produces another vector by computing $\sigma(Wx + b)$ — a matrix multiplication, plus a bias vector, passed through a non-linear function $\sigma$ like $\tanh$. Training means adjusting the matrices $W$ and biases $b$ so the network's output, on some training data, gets close to the target output. The objective is encoded as a *loss* $\mathcal{L}$ — for regression, typically the squared error between prediction and target. The standard recipe for adjustment is *gradient descent*: compute how the loss changes when each parameter is nudged (the *gradient*, $\partial\mathcal{L}/\partial\theta$), and step each parameter in the direction that decreases the loss. For stability, gradient descent is almost always done in *mini-batches*: rather than computing the gradient on one training sample at a time (noisy) or all of them at once (expensive), you average the gradient over a small batch of $K$ samples. The averaging is the bottleneck this post is about. Computing per-sample gradients in a fragment shader is straightforward — each pixel runs an independent forward and backward pass for its sample. *Averaging* them is the hard part, because averaging across pixels is exactly the cross-pixel communication that fragment shaders refuse to do. The shader fits a single 1D function $y(x) = \sin(8\pi x)\cos(x^2)\cos(16 x)\sin(-4 x)$, deliberately picked to be high-frequency enough that the network actually needs Fourier features [#Tancik2020] to fit it — a technique, popularized by NeRF [#Mildenhall2020], where the scalar input $x$ is first expanded into a vector of sines and cosines at increasing frequencies, giving the network something to grab onto besides a single smooth coordinate. The architecture is PE($L=6$) → 16 tanh → 16 tanh → 1 linear, for 497 parameters total, optimized with Adam [#Kingma2014] over a mini-batch of $K = 256$ samples drawn fresh every frame. # Why mipmaps A *mipmap* is something graphics programmers see every day: a precomputed pyramid of progressively shrunken copies of a texture. Level 0 is the original texture. Level 1 is the same image at half the resolution in each axis — each texel is the average of a 2×2 block of level-0 texels. Level 2 halves again, and so on. The reason mipmaps exist is anti-aliasing: when you render a faraway object, you want each on-screen pixel to sample a color that already represents an *average* over many texels, not a single point lookup that might fall on any color. The relevant observation for this shader is: **mipmap downsampling is, mathematically, an arithmetic mean.** Each texel at mipmap level $\ell$ is the average of the $2^\ell \times 2^\ell = 4^\ell$ texels under it at level 0. If we arrange Buffer C so that the $K = 4^\ell$ per-sample gradients for one parameter occupy exactly one $2^\ell \times 2^\ell$ block, then reading that block's location at mipmap level $\ell$ returns the *exact* mean of those gradients. The GPU has already done the averaging for us, in hardware, the moment it built the mipmap pyramid for that frame. In other words: we want $\bar{g}_p = \tfrac{1}{K}\sum_k g_{p,k}$. The graphics card, given a texture, computes the same kind of sum every time it generates a mipmap. The trick is just to convince the texture to hold the right values in the right places, so the GPU's automatic averaging produces the gradient mean as a side effect. # The pipeline Four passes cycle every frame. Buffer A holds the parameters and the Adam optimizer state, with $(w, m, v)$ — the weight, the first-moment estimate, and the second-moment estimate — packed into the RGB channels of a single texel per parameter. Buffer B draws $K = 256$ fresh training samples each frame, with each pixel storing one $(x, y)$ pair from a deterministic hash seeded by `iFrame`. Buffer C is the centerpiece: a 2D grid where each pixel stores a *single per-sample gradient for a single parameter*. The Image pass reads Buffer A and renders the network's current prediction over the target. Buffer C reads weights from A and samples from B and writes per-sample gradients. Buffer A reads averaged gradients from C and applies one Adam step. Buffer B refreshes the sample set. Image visualizes. # The mipmap trick, formally For each parameter $\theta_p$, what we want is the standard mini-batch gradient $$\bar{g}_p = \frac{1}{K} \sum_{k=1}^{K} \frac{\partial \mathcal{L}_k}{\partial \theta_p}.$$ Buffer C stores the $K$ per-sample gradients for parameter $\theta_p$ in a contiguous $K_X \times K_Y$ block of pixels, with $K_X = K_Y = 16$ and $K = K_X K_Y = 256$. The block for parameter $p$ sits at offset $(p_c \cdot K_X,\ p_r \cdot K_Y)$ in the texture, where $(p_c, p_r)$ is the parameter's 2D index in a row-major packing. Mipmapping with the default box filter produces, at mipmap level $\ell$ (in OpenGL terminology, *LOD* — Level Of Detail), $$T_\ell(p_c, p_r) = \frac{1}{4^\ell} \sum_{i=0}^{2^\ell - 1} \sum_{j=0}^{2^\ell - 1} T_0\!\left(2^\ell p_c + i,\ 2^\ell p_r + j\right).$$ Choose $\ell = \log_2 K_X = 4$. Then $2^\ell = 16$, $4^\ell = 256 = K$, and the right-hand side is exactly the arithmetic mean over parameter $p$'s 16×16 block — that is, $\bar{g}_p$. The whole reduction collapses into one line in Buffer A: ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~ glsl float g = texelFetch(iChannel1, ivec2(pCol, pRow), LOD_K).r; ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~ This is not an approximation. The equality is exact whenever three conditions hold: the block is filled with $K$ valid gradients (no zero-padding inside the block), the block is aligned to mipmap cell boundaries (offset divisible by $K_X$), and the texture sampler uses the default box filter for mip generation. The shader satisfies all three. The block offsets are integer multiples of 16 by construction, the active region is filled completely with 256 valid per-sample gradients, and Shadertoy generates mips with the default box filter. Pixels that fall outside the active grid of parameters are explicitly cleared to zero, which is harmless because they live in their own 16×16 blocks and are never sampled at LOD 4. # Pixel-wise gradient parallelism The other half of the design is how Buffer C gets filled. Each pixel decomposes its coordinate into a `(paramIdx, sampleIdx)` pair, fetches sample $k$ from Buffer B, runs a full forward + backward pass through the network using the current weights from Buffer A, and writes the single scalar gradient for parameter `paramIdx` at sample $k$: ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~ glsl int paramCol = p.x / K_X; int paramRow = p.y / K_Y; int paramIdx = paramRow * PARAMS_PER_ROW + paramCol; int kx = p.x - paramCol * K_X; int ky = p.y - paramRow * K_Y; int k = kx + ky * K_X; vec4 s = texelFetch(iChannel1, ivec2(kx, ky), 0); float x = s.r; float yT = s.g; float feat[IN_DIM], a1[H], a2[H], dz1[H], dz2[H]; float yhat, dy; forwardBackward(x, yT, iChannel0, feat, a1, a2, yhat, dz1, dz2, dy); float g = gradAtFlat(paramIdx, feat, a1, a2, dz1, dz2, dy); fragColor = vec4(g, 0, 0, 1); ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~ The honest cost is that every pixel re-runs the full forward and backward pass for its sample, just to extract one scalar gradient. A traditional implementation does one forward + backward per sample and reads all $N$ gradients from the cached activations, for $O(K)$ network evaluations. This shader does $O(K \cdot N)$ network evaluations — 256 × 497 ≈ 127k per frame. The trade is that there is no cross-pixel communication at all: every pixel is independent, and the GPU eats them in parallel. For a 497-parameter network this is comfortably under the per-frame budget. For a 100M-parameter model it obviously is not, but the shader is not pretending to be that. # The Adam update Buffer A is the only pass that uses the mipmap read. For each parameter pixel, it fetches its current $(w, m, v)$ at LOD 0, reads the averaged gradient $\bar{g}$ from Buffer C at LOD 4, and applies the standard Adam update [#Kingma2014]: $$m \leftarrow \beta_1 m + (1 - \beta_1) \bar{g}, \qquad v \leftarrow \beta_2 v + (1 - \beta_2) \bar{g}^2,$$ $$\hat{m} = \frac{m}{1 - \beta_1^t}, \quad \hat{v} = \frac{v}{1 - \beta_2^t}, \qquad w \leftarrow w - \eta\,\frac{\hat{m}}{\sqrt{\hat{v}} + \varepsilon}.$$ Here $m$ and $v$ are exponential moving averages of the gradient and its square (Adam's "moments"); $\hat{m}$ and $\hat{v}$ are their bias-corrected versions, important during the first few hundred steps when the moments are still warming up from zero; $\eta$ is the learning rate; $\beta_1, \beta_2, \varepsilon$ are the standard Adam constants. Weights are initialized once at `iFrame == 0` with Xavier-uniform scaling [#Glorot2010] using a per-pixel hash, which is enough for the network to start in a sensible regime without an external init pass. # Caveats A few constraints fall out of the design. Batch size must satisfy $K = 4^\ell$ for some integer $\ell$ — so 16, 64, 256, 1024, and so on — because that is the only family of box sizes a mip pyramid produces. The 8-bit byte storage Shadertoy uses by default limits gradient precision, which is fine for a small network but starts to bite once the loss approaches the noise floor; float textures would help. Any method with cross-sample dependencies, like BatchNorm, breaks the per-pixel independence assumption and would need extra passes. And the mipmap reduction itself is unweighted: importance-weighted samples would require encoding the weights into the gradients before they hit Buffer C. None of this is fatal. The point of the shader is to show that the standard backprop loop — forward, per-sample gradient, mini-batch average, optimizer step [#Rumelhart1986] — can run inside a fragment-shader pipeline with no extra reduction infrastructure, by noticing that the GPU already builds the reduction for you every time it samples a mipmap.
